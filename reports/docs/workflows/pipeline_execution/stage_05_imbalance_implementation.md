# Technical Implementation Report: Imbalance Handling Study

**Stage:** 05 — Imbalance Tuning  
**Focus:** Implementation Details ("The How")  
**Toolkit:** Imbalanced-Learn (imblearn) + Scikit-Learn

---

## 1. Overview
The Imbalance Handling implementation evaluates different resampling and reweighting strategies to mitigate class skew in the sentiment dataset. It operates as a comparison study where each method is applied to a fixed sparse feature matrix to observe its impact on minority class recall.

---

## 2. Technical Stack
- **Sampling Engine:** `imbalanced-learn`.
- **Feature Source:** `PF-IDF` (best params from Stages 3/4).
- **Experiment Tracking:** `mlflow`.
- **Benchmarking Model:** `RandomForestClassifier`.

---

## 3. Implementation Workflow

### 3.1 Sparse Resampling Logic
The implementation leverages the compatibility between `imblearn` and `scipy.sparse` matrices. Feature matrices generated by the `TfidfVectorizer` are passed directly to the samplers.

### 3.2 Strategy Implementation
The system implements five distinct handling paths:

1.  **Class Weights (`class_weights`):**
    - **Mechanism:** No synthetic data creation. Sets `class_weight='balanced'` in the RandomForest constructor.
    - **Advantage:** Low compute, zero memory overhead.

2.  **Synthetic Over-sampling (`oversampling` / SMOTE):**
    - **Mechanism:** Creates synthetic minority samples by interpolating between existing neighbors.
    - **Implementation:** `SMOTE(random_state=42)`.

3.  **Adaptive Over-sampling (`adasyn`):**
    - **Mechanism:** Similar to SMOTE but shifts the focus toward minority samples in "difficult" (higher density of majority samples) regions of the feature space.

4.  **Combinatorial Cleaning (`smote_enn`):**
    - **Mechanism:** Over-samples via SMOTE and then removes samples that are misclassified by their nearest neighbors.
    - **Advantage:** Cleans up noisy boundaries between neutral and negative sentiment.

### 3.3 Strict Partitioning Contract
A critical implementation detail is the **Resampling Boundary**:
- **Train Split:** Subject to `fit_resample()`.
- **Validation Split:** Subject to `transform()` ONLY.
The implementation ensures that the validation set distribution remains natively imbalanced, providing a realistic estimate of production performance.

---

## 4. Observability & Metadata

### 4.1 MLflow Experiment Design
Each resampling run is logged with detailed class-wise telemetry:
- **`imb_strategy`:** The specific algorithm name.
- **`resampled_size`:** The new dataset shape (often significantly larger than the original).
- **`val_f1_macro`:** The primary optimization target.

### 4.2 Figure Persistence
The implementation generates confusion matrices for every strategy and saves them to `reports/figures/imbalance_methods/`. These artifacts allow for manual auditing of whether a strategy correctly improved "Negative" recall at the cost of "Neutral" precision.

---

## 5. Robustness & Design Patterns

- **Deterministic Synthesis:** All imblearn samplers are instantiated with a fixed `random_state`. This ensures that synthetic point generation is reproducible across different compute environments.
- **Lazy Evaluation:** If a user specifies a strategy not in the `imblearn` library, the implementation provides descriptive error logging and fails before starting the expensive training loops.
- **Memory Efficiency:** Large sparse matrices are handled using `scipy.sparse`. The implementation avoids converting these to dense arrays, preventing memory-based crashes during over-sampling.

---

## 6. Execution Logic Summary
1.  **Initialize:** Fetch study configuration via `ConfigurationManager`.
2.  **Vectorize:** Generate the best TF-IDF feature matrix from Stages 3 and 4.
3.  **Loop:** For each configured `imbalance_method`:
    - Apply the specific resampling or reweighting logic.
    - Train the fixed-seed RandomForest model.
    - Log metrics and export Confusion Matrix PNGs.
4.  **Finalize:** Summarize the champion strategy in the logs and provide the MLflow Run ID.
