# Technical Implementation Report: Baseline Training Pipeline

**Stage:** 07 — Baseline Model  
**Focus:** Implementation Details ("The How")  
**Toolkit:** Scikit-Learn + MLflow + Joblib

---

## 1. Overview
The Baseline Training implementation establishes a predictive benchmark using a Logistic Regression architecture. It focuses on high-precision reproducibility, multi-environment model packaging (Model Bundle), and exhaustive classification telemetry across all sentiment labels.

---

## 2. Technical Stack
- **Estimator:** `LogisticRegression` (Liblinear solver).
- **Orchestration:** `mlflow` (Tracking API).
- **Serialization:** `pickle` (via `save_model_bundle`).
- **Data Loaders:** `scipy.sparse` (NPZ loading via `load_feature_data`).

---

## 3. Implementation Workflow

### 3.1 Synchronized Data Ingestion
The implementation uses a centralized `load_feature_data()` utility. This ensures that the training set and the corresponding `LabelEncoder` are loaded in correct sequence ($X_{train}, X_{val}, X_{test}, y_{train}, y_{val}, y_{test}, le$), preventing any mismatched label-feature indexing during the fit stage.

### 3.2 Imbalance-Aware Training
The logic leverages the `class_weight` parameter from the `LogisticBaselineConfig`. By using values like `"balanced"`, the implementation adjusts the weight of each class inversely proportional to its frequency in the training set, ensuring the model doesn't drift toward a "Majority Class Bias."

### 3.3 Label-Aware Evaluation
Prior to metric calculation, the implementation performs an **Inverse Transformation** of the predicted integer indices:
```python
y_val_orig = le.inverse_transform(y_val)
y_pred_val_orig = le.inverse_transform(y_pred_val)
```
This enables the logging of readable per-class F1-scores (e.g., `test_f1_Positive`) directly to MLflow, rather than opaque numbers (`test_f1_2`).

---

## 4. Artifact Security & Packaging

### 4.1 The Model Bundle Pattern
A core design feature is the **Atomic Bundle**. Instead of saving the estimator and encoder as separate files, they are packaged into a single dictionary:
```python
model_bundle = {
    "model": model,
    "encoder": le
}
```
This ensures that the inference microservice (FastAPI) always accesses a version of the text-to-label mapping that is mathematically synchronized with the trained weights.

### 4.2 DVC Metrics Integration
The implementation calls `save_baseline_metrics_json(score=val_f1)`. This produces a lightweight JSON file tracked as a `metric` in DVC, allowing developers to run `dvc metrics diff` to see performance changes across git commits without opening the MLflow UI.

---

## 5. Robustness & Design Patterns

- **Retry-Safe MLflow Context:** The implementation includes `mlflow.end_run()` before `start_run()`. This ensures that even if a previous orchestration step crashed, the baseline training begins with a clean, unpolluted MLflow context.
- **Strict Solver Configuration:** By explicitly naming `solver: "liblinear"` in the configuration, the implementation ensures stable convergence on the sparse feature matrices generated by TF-IDF.
- **Dimensionality Telemetry:** The implementation logs `feature_dim` as an MLflow parameter. This allows researchers to quickly diagnose performance drops caused by a sudden reduction in the vocabulary size during Stage 06.

---

## 6. Execution Logic Summary
1.  **Ingest:** Load sparse feature matrices and target arrays from `artifacts/feature_engineering/`.
2.  **Initialize:** Instantiate `LogisticRegression` using hyperparameters from `params.yaml`.
3.  **Train:** Fit the model on the weighted training labels.
4.  **Evaluate:** Generate macro-averaged F1 and per-class reports on Validation and Test sets.
5.  **Log:** Stream metrics and parameters to the MLflow tracking server.
6.  **Persist:** Export the `logistic_baseline.pkl` bundle and `baseline_metrics.json` for DVC and model deployment.
