# Technical Implementation Report: Automated Model Registration

**Stage:** 11 — Model Registration  
**Focus:** Implementation Details ("The How")  
**Toolkit:** MLflow Library + Python Client

---

## 1. Overview
The Model Registration implementation is the operational link between the experimental DAG (DVC) and the production serving layer. It ensures that only validated champions are indexed in the global model registry. It implements a non-interactive promotion policy based on the "Handover Contract" generated by the evaluation stage.

---

## 2. Technical Stack
- **API:** `mlflow` (Registry API).
- **Client:** `mlflow.tracking.MlflowClient`.
- **Logic:** Threshold-based gating + State transition.

---

## 3. Implementation Workflow

### 3.1 Handover Contract Parsing
The implementation starts by ingesting the `best_model_run_info.json`. This JSON provides two critical tokens:
- **`run_id`:** The unique identifier for the specific experiment run in MLflow.
- **`model_name`:** The algorithm name (e.g., `xgboost`).
The script uses these tokens to retrieve the actual metrics from the MLflow Tracking Server, verifying that the data in the registry matches the data in the local DVC state.

### 3.2 The Quality Gate Enforcement
Before any registration logic executes, the implementation performs a binary check:
```python
current_f1 = metrics.get("test_macro_f1", 0)
if current_f1 < config.f1_threshold:
    raise QualityGateError(f"Model F1 ({current_f1}) below threshold ({config.f1_threshold}).")
```
This hard-stop prevents sub-par models from being versioned, ensuring that the Registry only contains high-performing candidates.

### 3.3 Atomic Registration & Transition
The registration process occurs in two distinct API calls to ensure atomicity:
1.  **`mlflow.register_model()`:** Creates a new version (e.g., `Version 4`) under the named model entry (e.g., `youtube_sentiment_analysis`).
2.  **`transition_model_version_stage()`:** Uses the `MlflowClient` to programmatically move the newly created version to the **"Production"** stage.

---

## 4. Serving Transition & Rollback Readiness

### 4.1 Alias-Based Serving
By transitioning to the "Production" stage, the implementation enables "Alias-Based Serving." Downstream applications (like the FastAPI Inference Service) don't need to know the version number—they simply request the current "Production" model from the registry.

### 4.2 Automated Retraining Loop
This implementation is designed to support fully automated retrains. When the pipeline is triggered by new data:
- The winner is identified.
- The winner is registered as the *latest* version.
- The *latest* version replaces the *previous* version in the "Production" stage.
This creates a seamless Continuous Deployment (CD) loop for Machine Learning.

---

## 5. Robustness & Error Handling

- **Registry Existence Check:** The implementation checks if the model registry entry exists before attempting registration. If it doesn't (first run), it creates it automatically.
- **Run-Sync Verification:** If the `run_id` provided in the contract is not found in MLflow (e.g., due to a deleted experiment), the script fails loudly, preventing the registration of an "orphan" model.
- **MLflow Client Resiliency:** Individual API calls are wrapped in standard Python `try-except` blocks to handle temporary network timeouts to the tracking server.

---

## 6. Execution Logic Summary
1.  **Input:** Read `best_model_run_info.json`.
2.  **Verify:** Fetch metrics from MLflow and compare against the `f1_threshold`.
3.  **Register:** Link the `run_id` to a new version in the named Model Registry.
4.  **Promote:** Transition the new version to the `Production` stage.
5.  **Log:** Record the registration event in the local system logs for auditability.
